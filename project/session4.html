Estimation Techniques

Project estimation involves predicting the time, cost, and resources needed to complete project activities.

Traditional Estimation Approaches

Technique Description Accuracy Range
Guesstimating Informal, experience-based estimates ±50% or more
Delphi Technique Anonymous expert consensus through rounds of estimation ±30%
Analogous (Comparative) Based on similar past projects ±15%
Parametric Modeling Statistical relationship between variables ±10%
Bottom-Up Estimating individual tasks and summing up ±5%
1. Which estimation technique typically provides the most accurate results?



Correct Answer: Bottom-up estimation
Explanation: Bottom-up estimation is most accurate (±5%) because it involves estimating individual work packages and summing them up, though it's also the most time-consuming.

Software Engineering Estimation Techniques

Specialized techniques for estimating software development projects:

Function Point Analysis

Measures software size based on functionality delivered to users:

  • Inputs: Data entering the system
  • Outputs: Reports, screens, error messages
  • Inquiries: Interactive inputs requiring response
  • Logical Files: Internal data groups
  • Interfaces: External connections
Function Point Calculation:
FP = UAF × VAF
Where:
UAF = Unadjusted Function Points (sum of all weighted counts)
VAF = Value Adjustment Factor (0.65 + (0.01 × TDI))
TDI = Total Degree of Influence (sum of 14 system characteristics)

COCOMO (Constructive Cost Model)

Algorithmic model using Lines of Code (LOC) to estimate effort:

COCOMO Models

Type Description Effort Formula Duration Formula
Organic Small, simple projects PM = 2.4 × KDSI1.05 TDEV = 2.5 × PM0.38
Semi-Detached Medium complexity PM = 3.0 × KDSI1.12 TDEV = 2.5 × PM0.35
Embedded Complex, constrained PM = 3.6 × KDSI1.20 TDEV = 2.5 × PM0.32

Where: PM = Person Months, KDSI = Thousands of Delivered Source Instructions

Example: COCOMO Calculation

Project: Semi-Detached project with 10,600 Java LOC

  1. Convert LOC to Function Points: 10,600 LOC / 53 LOC per FP = 200 FP
  2. Calculate Effort: PM = 3.0 × (10.6)1.12 = 42.21 person-months
  3. Calculate Duration: TDEV = 2.5 × (42.21)0.35 = 9.26 months
  4. Calculate Team Size: People = PM / TDEV = 42.21 / 9.26 ≈ 4.55 (5 people)

Activity-on-Node (AON) Network Diagrams

Shows project activities as nodes with dependencies:

  • Forward Pass: Calculates Early Start (ES) and Early Finish (EF)
  • Backward Pass: Calculates Late Start (LS) and Late Finish (LF)
  • Float/Slack: LF - EF or LS - ES (total float for non-critical paths)
  • Critical Path: Longest path with zero float

PERT (Program Evaluation Review Technique)

Uses three estimates to account for uncertainty:

Expected Duration: (O + 4M + P) / 6
Standard Deviation: (P - O) / 6
Variance: [(P - O) / 6]2
Where:
O = Optimistic, M = Most Likely, P = Pessimistic

Advanced Scheduling Concepts

Dependency Relationships

Type Symbol Description Example
Finish-to-Start (FS) → B can't start until A finishes Pour concrete after foundation is dug
Start-to-Start (SS) ⇉ B can't start until A starts Level concrete while pouring begins
Finish-to-Finish (FF) ⇶ B can't finish until A finishes Inspect wiring while installation completes
Start-to-Finish (SF) ⇝ B can't finish until A starts Roof assembly completes after truss delivery begins

Lead and Lag Times

  • Lead: Overlapping activities (starting B before A finishes)
  • Lag: Waiting time between activities (paint drying before carpeting)

Critical Chain Method

Addresses Parkinson's Law (work expands to fill time) by:

  • Aggregating safety time at the end (project buffer)
  • Using feeding buffers for non-critical paths
  • Focusing on resource constraints

Project Cost Management

Budget Components

Component Description Example
Direct Costs Directly attributable to project work Labor, materials, equipment
Indirect Costs Overhead shared across projects Office space, utilities
Contingency Reserve Known risks (within project scope) 10% of estimated costs
Management Reserve Unknown risks (outside project scope) 5% of total budget

Learning Curves

As workers gain experience, time per unit decreases:

Learning Curve Formula:
Yx = aXb
Where:
Yx = Time for xth unit
a = Time for 1st unit
X = Unit number
b = log(learning %)/log(2) (e.g., 0.322 for 80% learning curve)
2. What is the correct formula for calculating expected duration in PERT analysis?



Correct Answer: (O + 4M + P) / 6
Explanation: PERT uses weighted average that gives 4x weight to the most likely estimate (M) compared to optimistic (O) and pessimistic (P) estimates.

Key Takeaways

  • Estimation accuracy improves from guesstimating (±50%) to bottom-up (±5%)
  • Function Points measure software size based on user functionality
  • COCOMO estimates effort and duration based on lines of code
  • Gantt charts visualize schedule, while AON diagrams show dependencies
  • Critical path is the longest path with zero float
  • PERT uses three estimates to account for uncertainty
  • Budget includes direct costs, indirect costs, and reserves
  • Learning curves show productivity improvements over time
