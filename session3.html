📚 Session 3: How to Earn Software Quality

Quality Assurance, Quality Control, and Testing Methodologies

🔍 Quality Concepts Recap

What is Quality?

Quality is multifaceted and can be viewed from different perspectives:

  • Judgment-based: Based on evaluations by individuals or organizations
  • Fitness for purpose: Does it fulfill its intended function?
  • Rating/Measurement: Can be quantified against standards
  • Performance: Meets acceptable performance criteria
  • Requirements: Satisfies specified goals and requirements

Why Quality Matters

Quality is the ability of your product to satisfy users. Without quality, your product has little value:

Example: A banking app that frequently crashes or shows incorrect balances would fail in its fundamental purpose, regardless of how many features it has.

Quality helps future-proof projects by ensuring they remain maintainable, scalable, and adaptable to changing requirements.

🛡️ Quality Assurance Process

What is Quality Assurance?

Quality Assurance is the process that demonstrates your product can satisfy users. It focuses on preventing defects rather than finding them.

Example: Implementing coding standards, conducting training sessions, and establishing review processes are QA activities that prevent defects from occurring in the first place.

Benefits of Good QA

  • Improvements in usability and performance
  • Reduced defect rates
  • More predictable development processes
  • Higher customer satisfaction
  • Lower long-term maintenance costs

QA Requirements

Effective QA requires documented standards and best practices:

Establish Standards
Define Processes
Train Team

Continuous Improvement

💡 Remember:

Quality and best practice should always be considered in terms of being 'Fit for Purpose' - what's appropriate for one project may not be for another.

QA in the Software Development Lifecycle

QA activities should be integrated throughout the entire development process, not just at the end.

🔎 Quality Control Process

What is Quality Control?

Quality Control is the implementation of regular testing procedures against your quality definitions and the refinement of these procedures.

Example: Executing test cases, logging defects, and verifying fixes are all QC activities that help identify and eliminate defects in the product.

Key Aspects of QC

  • Formal use of testing: Systematic, documented testing procedures
  • Acting on results: Not just finding bugs, but ensuring they're fixed
  • Planning: Structured test plans with clear objectives
  • Documentation: Clear records of tests and results
  • Standards: Following established testing standards

Detection vs. Prevention

Aspect QA (Prevention) QC (Detection)
Focus Establish suitable development environment Inspections, audits, reviews
Activities Define processes and procedures Unit, integration, system testing
Measurement Measure process effectiveness Acceptance testing
Goal Continuous improvement Identify defects

Quality Circle

QC relates to the output and forms part of a continuous improvement cycle.

🧪 Software Testing Types

Testing in the SDLC

Testing should be aligned with different stages of the Software Development Life Cycle:

Testing Approaches

Testing can be categorized based on how it's performed:

Static Testing
Dynamic Testing

Static Testing

  • Hand execution: Reading and reviewing source code
  • Walk-Throughs: Informal presentations to peers
  • Code Inspections: Formal reviews with checklists
  • Automated Tools: Checking for syntax errors and standards compliance

Dynamic Testing

  • Black-box testing: Testing input/output behavior without knowing internal workings
  • White-box testing: Testing internal logic and code paths

Detailed Testing Types

Unit Testing

Purpose: Verify individual components (units) work correctly in isolation
Performed by: Developers
Example: Testing a single function that calculates interest rates

Integration Testing

Purpose: Verify interfaces between components work correctly
Performed by: Developers
Example: Testing how the payment processing module interacts with the user account module

System Testing

Purpose: Verify the complete system meets requirements
Performed by: QA team
Example: End-to-end testing of an e-commerce checkout process

Acceptance Testing

Purpose: Verify the system meets business requirements and is ready for use
Performed by: Clients/business users
Example: Real-world testing of a new banking feature by bank staff

⚠️ Dealing with Errors and Faults

Error Terminology

Before dealing with errors, we need to understand the terminology:

Failure: Observable incorrect behavior
Error: Erroneous state in the system
Fault: Underlying cause (bug) in the code

💡 Important:

Before identifying errors, we must first specify the desired behavior clearly!

Types of Faults

Algorithmic Fault

Example: A sorting algorithm that doesn't handle edge cases correctly, like an empty input array.

Mechanical Fault

Example: A disk write operation that fails due to hardware issues.

Approaches to Dealing with Errors

Verification
Modular Redundancy
Declare as Feature
Patching
Testing

Evaluation of Approaches

Approach Pros Cons
Verification Catches issues early May omit important constraints
Modular Redundancy Provides fault tolerance Expensive to implement
Declare as Feature Quick "fix" Bad practice, leads to technical debt
Patching Immediate solution Slows down performance, temporary fix
Testing Systematic approach Never complete (can't prove absence of bugs)

Comprehensive Error Management

Error Prevention (Before Release)

  • Good programming methodologies to reduce complexity
  • Version control to prevent inconsistencies
  • Verification to catch algorithmic bugs

Error Detection (During Operation)

  • Testing: Create failures in a planned way
  • Debugging: Start with unplanned failures
  • Monitoring: Find performance issues

Error Recovery (After Failure)

  • Database transaction rollbacks
  • Modular redundancy
  • Recovery blocks

Key Testing Observations

1. It is impossible to completely test any module or system

2. Testing can only show the presence of bugs, not their absence (Dijkstra)

3. Testing requires creativity and deep system understanding

4. Independent testers often find issues developers miss

🚀 Acceptance Test Driven Development (ATDD)

What is ATDD?

ATDD is an agile development approach that emphasizes collaboration between customers, developers, and testers to validate that we're "building the right code."

Customer
Developer
Tester

Key characteristics:

  • Express requirements as executable specifications
  • Create tests before writing implementation code
  • Ensure tests are meaningful to all stakeholders

Why We Need ATDD

Traditional requirements often lead to misunderstandings that are discovered too late:

Customer Requirements

1. Have one trunk

2. Have four legs

3. Carry passengers & cargo

4. Black in color

5. Herbivorous

Developer Solution

✅ 1. Have one trunk

✅ 2. Have four legs

✅ 3. Carry passengers & cargo

✅ 4. Black in color

✅ 5. Herbivorous

Bonus: Also gives milk!

🐄

Actual Need

Customer wanted an elephant 🐘 for safari tours, not a cow for dairy!

ATDD would have revealed this misunderstanding early through concrete examples and executable specifications.

Problems ATDD Solves:

  • Cumulative effects of small misunderstandings
  • Requirements lost in translation
  • Perception gaps between stakeholders
  • Late discovery of flawed requirements

ATDD Process

1. Collaborate to define acceptance criteria
2. Automate criteria as tests
3. Develop to pass tests
4. Refine based on feedback

Testing Activities Throughout SDLC

Testing should be integrated throughout the entire development lifecycle, from unit tests by developers to acceptance tests by users.

🧠 Practice Quiz

Test Your Understanding:

Q1: What's the difference between Quality Assurance and Quality Control?
Answer: QA focuses on preventing defects through process improvement, while QC focuses on detecting defects through testing and inspection. QA is process-oriented and spans the entire lifecycle, while QC is product-oriented and typically occurs during testing phases.
Q2: What are the three main categories in McCall's Quality Factor Model?
Answer: 1) Product Operation (correctness, reliability, etc.), 2) Product Revision (maintainability, flexibility), and 3) Product Transition (portability, reusability).
Q3: What are the four main levels of testing in the testing pyramid?
Answer: From bottom to top: 1) Unit testing, 2) Integration testing, 3) System testing, and 4) Acceptance testing.
Q4: What's the difference between static and dynamic testing?
Answer: Static testing involves examining code without executing it (reviews, walkthroughs), while dynamic testing involves executing the code with various inputs to observe behavior.
Q5: What are three approaches to dealing with software errors?
Answer: Possible answers include: 1) Verification, 2) Modular redundancy, 3) Declaring as feature, 4) Patching, 5) Testing, 6) Error prevention techniques, 7) Monitoring.
Q6: What problem does ATDD solve, and how?
Answer: ATDD solves misunderstandings in requirements by bringing customers, developers and testers together to create executable specifications before development begins. This ensures everyone shares the same understanding of what needs to be built.
