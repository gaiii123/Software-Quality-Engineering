üöÄ Introduction to Automation Testing

What is Automation Testing?

Automation testing is the process of using specialized tools and scripts to execute test cases automatically, compare actual outcomes with expected results, and generate detailed test reports without human intervention.

Key Characteristics:

  • Repeatable: Tests can be run multiple times with consistent results
  • Programmable: Test scenarios are coded as scripts
  • Comprehensive: Can test thousands of scenarios quickly
  • Reliable: Eliminates human errors in repetitive tasks
  • Fast: Executes much faster than manual testing

Why Automate Testing?

Benefits:
  • Efficiency: Run tests 24/7 without human oversight
  • Coverage: Execute thousands of complex test cases in minutes
  • Accuracy: Eliminate human errors in repetitive tasks
  • Reusability: Test scripts can be reused across versions
  • Cost-effective: Saves money in long-term despite initial investment
When to Automate:
  • Repetitive test cases
  • Complex calculations
  • Large data volume testing
  • Performance/load testing
  • Regression testing

üí° Automation Testing ROI

While automation requires initial investment in tools and training, it provides significant returns:

  • 70-80% reduction in regression testing time
  • Up to 90% reduction in repetitive test execution
  • 300-500% increase in test coverage
  • Early bug detection reduces fixing costs

1. Functional Test Automation

Examples:
  • Login functionality

  • Form validations

  • Security vulnerability scanning
  • Cross-browser compatibility
  • Accessibility testing

Classification by Test Phase

1. Unit Test Automation

Tests individual components/functions in isolation

Example: Testing a currency conversion function with various inputs

  expect(convertCurrency(100, 'USD', 'EUR')).toBe(85);

Tests application programming interfaces

Example: Testing a REST API endpoint 
// Postman test script example

    pm.expect(pm.response.responseTime).to.be.below(200);

Tests graphical user interface and user interactions

Example: Testing a login page 
// Selenium WebDriver example in Java

üí° Automation Pyramid Strategy

Ideal test automation distribution:

  • 70% Unit Tests (fast, cheap to maintain)
  • 20% API Tests (stable, less flaky than UI)
  • 10% UI Tests (expensive but necessary)

üìã Automation Testing Process

Implementing Test Automation

A structured approach ensures successful automation implementation:

1. Define Scope
2. Select Tools
3. Design Framework
4. Develop Scripts
5. Execute Tests
6. Maintain

1. Define Automation Scope

Identify what to automate based on:

  • ROI Potential: High repetition, critical features
  • Technical Feasibility: Stable UI elements
  • Business Criticality: Core functionalities
  • Test Data Needs: Large data volume scenarios

2. Select Appropriate Tools

Choose tools based on:

  • Application technology stack
  • Team skills and expertise
  • Budget constraints
  • Integration requirements
  • Reporting needs

3. Design Automation Framework

Key framework components:

  • Data-Driven: External test data sources
  • Modular: Reusable components
  • Keyword-Driven: Action keywords
  • Hybrid: Combination of approaches

4. Develop and Execute Test Scripts

Implementation best practices:

  • Follow coding standards
  • Implement proper waits and synchronizations
  • Use page object model for UI tests
  • Include meaningful assertions
  • Integrate with CI/CD pipeline

5. Maintain Automation Suite

Ongoing maintenance activities:

  • Regular script reviews and refactoring
  • Update for application changes
  • Handle flaky tests
  • Expand coverage incrementally
  • Monitor and improve execution speed

üí° Framework Best Practices

  • Implement proper error handling
  • Use explicit waits instead of sleeps
  • Separate test data from scripts
  • Generate detailed reports
  • Version control your test code

Calculator Test Automation

From the lecture slides expanded with additional context:

Manual Test Case

  1. Launch Calculator
  2. Press 2
  3. Press +
  4. Press 3
  5. Press =
  6. Verify screen shows 5
  7. Close Calculator

Selenium WebDriver Implementation (Java)

import org.openqa.selenium.By;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import org.testng.Assert;
import org.testng.annotations.Test;

public class CalculatorTest {
    
    @Test
    public void testAddition() {
        // Set ChromeDriver path
        System.setProperty("webdriver.chrome.driver", "path/to/chromedriver");
        
        // Initialize WebDriver
        WebDriver driver = new ChromeDriver();
        
        try {
            // Navigate to online calculator
            driver.get("https://www.online-calculator.com/");
            
            // Perform calculation
            driver.findElement(By.name("two")).click();
            driver.findElement(By.name("plus")).click();
            driver.findElement(By.name("three")).click();
            driver.findElement(By.name("equals")).click();
            
            // Get result
            String result = driver.findElement(By.id("display")).getAttribute("value");
            
            // Assertion
            Assert.assertEquals(result, "5", "Addition calculation failed");
            
        } finally {
            // Close browser
            driver.quit();
        }
    }
}

API Test Automation Example

Testing a weather API with Postman:

Postman Test Script

// Set environment variables
pm.environment.set("apiKey", "your_api_key");
pm.environment.set("baseUrl", "https://api.weatherapi.com/v1");

// Test current weather endpoint
pm.test("Current weather returns valid data", function() {
    pm.sendRequest({
        url: pm.environment.get("baseUrl") + "/current.json?key=" + 
             pm.environment.get("apiKey") + "&q=London",
        method: 'GET'
    }, function(err, res) {
        // Check status code
        pm.expect(res.code).to.equal(200);
        
        // Check response time
        pm.expect(res.responseTime).to.be.below(500);
        
        // Check response structure
        pm.expect(res.json()).to.have.property('location');
        pm.expect(res.json().location).to.have.property('name', 'London');
        pm.expect(res.json()).to.have.property('current');
    });
});

Unit Test Example

Testing a simple string utility function:

JavaScript Unit Test with Jest

// stringUtils.js
function reverseString(str) {
    if (typeof str !== 'string') {
        throw new Error('Input must be a string');
    }
    return str.split('').reverse().join('');
}

module.exports = reverseString;

// stringUtils.test.js
const reverseString = require('./stringUtils');

test('reverses a string correctly', () => {
    expect(reverseString('hello')).toBe('olleh');
});

test('reverses an empty string', () => {
    expect(reverseString('')).toBe('');
});

test('throws error for non-string input', () => {
    expect(() => reverseString(123)).toThrow('Input must be a string');
});

üí° Assertion Best Practices

  • Make assertions specific and meaningful
  • Include descriptive failure messages
  • Verify both positive and negative scenarios
  • Test edge cases and boundary values
  • Keep assertions atomic (one verification per test)

üß† Automation Testing Quiz

Test Your Knowledge:

Q1: What are the three main types of automation tests based on test phase?
Answer: Unit tests, API tests, and UI tests.
Q2: Why is the "Pesticide Paradox" relevant to test automation?
Answer: The Pesticide Paradox states that if the same tests are repeated continuously, they'll eventually stop finding new bugs. In automation, this means we need to regularly review and update our test cases to find different defects.
Q3: What are the key components of a good test automation framework?
Answer: Key components include:
  • Test data management
  • Reporting mechanism
  • Error handling
  • Reusable libraries
  • Configuration management
  • CI/CD integration
Q4: What's the difference between functional and non-functional test automation?
Answer: Functional automation tests what the system does (features and business logic), while non-functional automation tests how the system performs (performance, security, usability, etc.).
Q5: What are some signs that a test case is a good candidate for automation?
Answer: Good candidates for automation are:
  • Repetitive test cases
  • Tests that run on multiple configurations
  • Tests requiring large data volumes
  • Frequently executed regression tests
  • Tests that are difficult to perform manually
  • Business-critical functionalities
Q6: What are the key differences between Selenium and Cypress for UI automation?
Answer: Key differences:
  • Architecture: Selenium works through WebDriver protocol, Cypress runs in the browser
  • Speed: Cypress is generally faster
  • Debugging: Cypress has better debugging tools
  • Language: Selenium supports multiple languages, Cypress is JavaScript only
  • Cross-browser: Selenium supports more browsers
  • Learning Curve: Cypress is easier to learn
