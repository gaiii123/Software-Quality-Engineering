üß≠ Seven Fundamental Testing Principles

Why Testing Principles Matter

These principles form the foundation of effective software testing and guide testers in making decisions about test planning, design, and execution.

1. Exhaustive Testing is Impossible

It's impractical to test all combinations of inputs and preconditions.

Example: A login form with username (any 8-char string) and password (any 8-char string) has 26^8 √ó 26^8 = 208,827,064,576 possible combinations. Testing all would take centuries!

Solution: Use equivalence partitioning and boundary value analysis to select representative test cases.

2. Defect Clustering

A small number of modules usually contain most of the defects.

Example: In a banking app, 80% of bugs might be found in the interest calculation module (20% of code).

Solution: Focus testing on complex or frequently changed modules.

3. Pesticide Paradox

Repeating the same tests will stop finding new bugs.

Example: If you always test a form with the same values, you'll miss bugs that only appear with unusual inputs.

Solution: Regularly review and update test cases. Use exploratory testing.

4. Testing Shows Presence of Defects

Testing can prove defects exist, but cannot prove there are no defects.

Example: You find 50 bugs in testing, but there might still be 50 more undiscovered.

Solution: Use risk analysis to determine when testing is "good enough."

5. Absence of Errors - Fallacy

Finding and fixing defects doesn't help if the system is unusable.

Example: A bug-free accounting system that doesn't meet user needs is worthless.

Solution: Ensure testing validates requirements and user needs, not just technical correctness.

6. Early Testing

Testing should start as early as possible in the SDLC.

Example: Finding requirements ambiguities during review prevents costly rework later.

Solution: Implement shift-left testing with requirements and design reviews.

7. Testing is Context Dependent

Testing approaches differ based on the project context.

Example: Medical device software requires more rigorous testing than a simple mobile game.

Solution: Adapt testing strategy based on risk, domain, and project constraints.

üí° Practical Application:

These principles aren't just theoretical - they directly impact how we:

  • Prioritize test cases (focus on high-risk areas)
  • Allocate testing resources (defect clustering)
  • Determine when to stop testing (exhaustive testing impossible)
  • Communicate test results (testing shows presence of defects)

üîç Test Types and Quality Characteristics

What are Test Types?

Groups of test activities aimed at testing specific characteristics of a software system based on specific objectives.

Four Main Test Types:

1. Functional Testing

Evaluates functional quality characteristics like completeness, correctness, and appropriateness.

Examples:
  • Does the login function authenticate users correctly?
  • Does the shopping cart calculate totals accurately?

2. Non-Functional Testing

Evaluates characteristics like reliability, performance, security, and usability.

Examples:
  • Can the system handle 1000 concurrent users?
  • Is sensitive data properly encrypted?

3. Structural Testing

Evaluates the architecture and implementation of the system.

Examples:
  • Is all code exercised by tests? (code coverage)
  • Are database connections properly closed?

4. Change-Related Testing

Evaluates effects of changes through confirmation and regression testing.

Examples:
  • Does the bug fix resolve the reported issue?
  • Did the new feature break existing functionality?

Banking Application Examples Across Test Levels:

Functional Testing Example

Component Testing

Test interest calculation component with various principal amounts, rates, and time periods.

System Testing

Test end-to-end credit application workflow from customer submission to approval.

Acceptance Testing

Verify bankers can properly approve/decline applications based on policy rules.

Non-Functional Testing Example

Component Testing

Measure CPU usage during complex interest calculations.

System Testing

Verify UI works on all supported browsers and devices.

Acceptance Testing

Evaluate interface accessibility for users with disabilities.

üí° Key Quality Characteristics:

  • Usability: How easy is the system to use?
  • Reliability: Does it perform consistently?
  • Security: Is data protected?
  • Efficiency: Does it use resources well?
  • Maintainability: Is it easy to modify?

Three Main Approaches to Testing

Different techniques are used based on what information is available and what aspects need testing.

1. Black-Box Techniques

Test the system without knowledge of internal implementation.

Characteristics:
  • Based on requirements and specifications
  • Tests functionality from user perspective
  • Examples: Equivalence partitioning, boundary value analysis, decision tables
Example: Testing a login screen by trying valid/invalid credentials without knowing how authentication is implemented.

2. White-Box Techniques

Test internal structures and implementation.

Characteristics:
  • Requires knowledge of code/internal design
  • Tests paths, conditions, and data flows
  • Examples: Statement coverage, branch coverage, path testing
Example: Ensuring all branches in a complex interest calculation function are executed during testing.

3. Experience-Based Techniques

Leverage tester knowledge and intuition.

Characteristics:
  • Based on tester expertise and domain knowledge
  • Often used to complement other techniques
  • Examples: Error guessing, exploratory testing, checklist-based testing
Example: A tester familiar with banking systems knows to test for rounding errors in financial calculations.

üí° Technique Selection Factors:

  • System complexity and risk level
  • Available documentation and specifications
  • Tester knowledge and skills
  • Testing objectives and constraints
  • SDLC model being used

üìã Test Process: Activities and Tasks

The Testing Lifecycle

A structured approach to testing ensures comprehensive coverage and efficient use of resources.

Test Planning
Test Monitoring
Test Analysis
Test Design
Test Implementation
Test Execution
Test Completion

1. Test Planning

Define objectives, approach, resources, and schedule.

Activities:
  • Determine test scope and strategy
  • Identify required resources
  • Define entry and exit criteria
  • Create test plan document

2. Test Monitoring and Control

Track progress and make adjustments as needed.

Activities:
  • Compare actual progress to plan
  • Report test status
  • Adjust test activities based on results

3. Test Analysis

Determine "what to test" by identifying test conditions.

Activities:
  • Review test basis (requirements, etc.)
  • Identify testable features
  • Define test conditions (things to verify)

4. Test Design

Determine "how to test" by creating test cases.

Activities:
  • Design high-level test cases
  • Identify test data requirements
  • Define test preconditions

5. Test Implementation

Prepare for test execution.

Activities:
  • Develop and prioritize test cases
  • Create test data and scripts
  • Set up test environment

6. Test Execution

Run tests and report results.

Activities:
  • Execute tests according to plan
  • Compare actual vs expected results
  • Report defects
  • Log test results

7. Test Completion

Wrap up testing activities.

Activities:
  • Evaluate exit criteria
  • Create test summary report
  • Archive test assets
  • Identify lessons learned

üí° Process Adaptation:

The test process should be adapted based on:

  • Project context and constraints
  • Software development lifecycle model
  • Product and project risks
  • Available resources

üß† Psychology of Testing

The Human Element in Testing

Effective testing requires understanding human psychology and fostering positive relationships between testers and other team members.

Common Psychological Challenges

  • Testing as Criticism: Developers may perceive bug reports as personal criticism
  • Quality Ownership: The "throw it over the wall" mentality between dev and test
  • Defensive Reactions: Resistance to reported defects
  • Priority Conflicts: Business vs quality priorities

Strategies for Positive Testing Culture

1. Collaborative Approach:
  • Frame testing as a team effort toward quality
  • Use "we" language instead of "you" when discussing issues
  • Involve developers in test planning
2. Constructive Communication:
  • Report defects factually without blame
  • Focus on the problem, not the person
  • Provide clear reproduction steps
3. Emphasize Benefits:
  • Highlight how testing saves time/money in long run
  • Show how defects improve product quality
  • Celebrate quality improvements
4. Understand Perspectives:
  • Recognize developer pride in their work
  • Acknowledge business pressures
  • Find win-win solutions

Effective Defect Reporting

Do:
  • Provide clear, concise steps to reproduce
  • Include screenshots or logs when helpful
  • State expected vs actual behavior
  • Suggest severity/priority (but be open to discussion)
Don't:
  • Use emotional or accusatory language
  • Assume developer's intentions
  • Report duplicates without checking
  • Ignore the business context

üí° Building Quality Culture:

Quality is everyone's responsibility. Foster an environment where:

  • Defects are seen as opportunities to improve
  • Testers are valued as quality advocates
  • Developers welcome feedback
  • Business stakeholders understand quality tradeoffs

Estimating Testing Effort

Accurate test estimation helps with resource planning and project scheduling.

1. Metrics-Based Technique

Use historical data from similar projects to estimate current effort.

Approaches:
  • Test Case Count: Average time per test case √ó number of test cases
  • Defect Density: Expected defects √ó average time to find/fix
  • Burndown Charts: Track remaining work over time
Example: If previous project had 500 test cases taking 200 hours, and current project has 750 test cases, estimate 300 hours (750/500 √ó 200).

2. Expert-Based Technique

Rely on experience of testing experts to estimate effort.

Approaches:
  • Delphi Technique: Multiple experts provide estimates anonymously, then discuss
  • Work Breakdown: Decompose testing into tasks and estimate each
  • Analogous Estimating: Compare to similar past projects
Example: Senior tester estimates 3 weeks for API testing based on similar project last year.

Factors Affecting Test Estimation

  • Product Characteristics: Complexity, quality requirements
  • Development Process: Agile vs waterfall, documentation quality
  • Test Team: Experience, skills, availability
  • Test Environment: Availability, stability
  • Tools: Automation level, reporting needs
  • Risks: High-risk areas needing more testing

üí° Estimation Best Practices:

  • Use multiple techniques and compare results
  • Include buffer time for unexpected issues
  • Re-estimate regularly as project evolves
  • Track actuals vs estimates to improve future accuracy
  • Document assumptions behind estimates

üß† Practice Quiz

Test Your Understanding:

Q1: Which testing principle explains why we should regularly update our test cases?
Answer: The Pesticide Paradox principle states that repeating the same tests will stop finding new bugs, so we need to regularly review and update our test cases.
Q2: What's the difference between functional and non-functional testing?
Answer: Functional testing verifies what the system does (features and functions), while non-functional testing evaluates how the system performs (characteristics like performance, security, usability).
Q3: Name three strategies for effective defect communication.
Answer: Any three of:
  • Report defects factually without blame
  • Provide clear reproduction steps
  • Focus on the problem, not the person
  • Use collaborative "we" language
  • Include expected vs actual behavior
Q4: What are the main advantages of metrics-based test estimation?
Answer: Advantages include:
  • Based on historical data (more objective)
  • Can be more accurate for similar projects
  • Provides quantitative basis for estimates
  • Easier to justify to stakeholders
Q5: Match these test activities to their correct phase in the test process:

a) Create test summary report
b) Design high-level test cases
c) Determine test scope and strategy
d) Compare actual vs expected results

Answer:
  • a) Test Completion
  • b) Test Design
  • c) Test Planning
  • d) Test Execution
Q6: Why is early testing important, and what activities can be done early in the SDLC?
Answer: Early testing is important because defects found earlier are cheaper to fix. Early activities include:
  • Requirements reviews
  • Design document analysis
  • Test planning
  • Early prototype testing
  • Static code analysis
